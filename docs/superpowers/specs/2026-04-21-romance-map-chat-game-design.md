# 现代恋爱向地图探索对话游戏设计文档

## 1. 概述

本项目第一版要做的是一个手机端竖屏网页游戏，核心体验是“地图探索进入事件，自然语言推进恋爱剧情”。玩家先在城市地图中选择区域，再进入一层子场景，最后在具体地点触发剧情事件。进入事件后，玩家通过自然语言与角色互动，由大模型生成剧情推进内容。

项目目标不是做一个纯聊天工具，也不是做一个资源管理游戏，而是做一个具有明确“地点进入感”和“剧情持续性”的恋爱冒险游戏原型。

## 2. 第一版设计目标

- 做出手机竖屏专用的网页游戏体验
- 提供“世界地图 → 子场景 → 剧情事件 → 自然语言对话”的完整闭环
- 让上下文在整局游戏中持续保留，不因切换地点或退出事件而重置
- 提供玩家手动触发的上下文压缩能力，产出短摘要与关键记忆
- 验证恋爱题材下，自然语言互动与轻量探索机制能否形成稳定沉浸感

## 3. 非目标

第一版不包含以下内容：

- 时间、体力、金钱等硬资源系统
- 明确失败条件或高压惩罚机制
- 多层复杂地图探索
- 多名可攻略主角色并行的复杂关系系统
- 自动上下文压缩
- 数值化好感度系统
- 完整历史档案、任务系统、背包系统等扩展玩法

## 4. 目标平台与视觉布局

### 4.1 目标平台

- 仅支持手机端竖屏比例
- 第一版不专门适配桌面端复杂布局

### 4.2 屏幕结构

屏幕分为上下两部分：

- 上半屏约占 60% 到 65%，负责显示主视觉内容
- 下半屏约占 35% 到 40%，负责显示文本、按钮、输入框和状态信息

### 4.3 上半屏职责

根据不同阶段显示不同内容：

- 地图层：显示城市总地图
- 子场景层：显示区域内场景图
- 事件层：显示角色立绘、场景插图或地点背景

### 4.4 下半屏职责

下半屏固定为文字冒险式交互区域，包含以下子区块：

1. 顶部状态条
   - 显示当前位置，例如“学校 / 教室”
   - 显示当前模式，例如“探索中”或“事件中”
   - 提供“记忆压缩”按钮
   - 提供“返回上一级”按钮
2. 文本显示区
   - 展示旁白、角色发言、系统提示
3. 操作区
   - 探索时显示地点按钮、进入确认等操作
   - 事件时显示少量辅助操作，例如“结束事件”
4. 输入区
   - 事件阶段启用
   - 提供多行输入框与发送按钮

## 5. 核心玩法循环

第一版的主循环如下：

1. 玩家进入城市总地图
2. 玩家点击一个主区域
3. 玩家进入该区域的一层子场景列表
4. 玩家点击某个子场景
5. 若该子场景存在剧情事件，则在下半屏显示进入提示
6. 玩家确认后进入剧情事件
7. 玩家通过自然语言与角色互动，推动剧情发展
8. 事件结束后返回地点或区域，世界状态保留
9. 玩家继续探索其他地点或再次进入已有地点

该循环的设计原则如下：

- 探索负责决定故事发生地点
- 事件负责承接具体戏剧场面
- 自然语言输入负责决定玩家如何介入人物关系

## 6. 世界地图与场景结构

### 6.1 世界地图

第一版只做一张城市总地图，包含四个主区域：

- 学校
- 医院
- 商场
- 主角家

### 6.2 子场景结构

每个主区域只做一层子场景探索。

#### 学校

- 教室
- 走廊
- 操场
- 天台

#### 医院

- 大厅
- 病房
- 走廊
- 自动贩卖机区

#### 商场

- 一层中庭
- 咖啡店
- 电影院门口
- 饰品店

#### 主角家

- 客厅
- 卧室
- 阳台
- 门口

### 6.3 设计原则

- 地图结构保持简单，避免玩家迷路
- 一层子场景足以提供“进入感”，同时控制实现成本
- 地点命名与视觉表达要服务恋爱剧情气质，而不是强调复杂玩法

## 7. 剧情事件结构

### 7.1 事件类型

第一版支持三类基础事件：

1. 固定事件
   - 某个地点首次进入时触发
2. 重复可触发事件
   - 某些地点可重复出现短互动
3. 状态延续事件
   - 某个前序事件完成后，在另一个地点解锁后续剧情

### 7.2 事件数据字段

每个事件至少包含以下字段：

- 事件标题
- 发生地点
- 出场角色
- 触发条件
- 开场文本
- 事件目标
- 结束条件
- 结束后的世界变化

### 7.3 第一版内容规模

第一版建议内容范围如下：

- 1 名核心恋爱对象
- 1 名辅助角色
- 8 个左右剧情事件
- 1 条可从头到尾完整跑通的恋爱短线

### 7.4 内容重心

- 学校与商场承担主线推进任务
- 医院与主角家承担情绪延展、补充信息和私密剧情任务

## 8. 对话驱动设计

### 8.1 输入方式

事件阶段采用纯自然语言输入。玩家可以输入：

- 对角色说的话
- 玩家的行动描述
- 态度表达或追问内容

### 8.2 事件层目标

事件中的对话不是无限闲聊，而是围绕当前场景目标推进。例如：

- 初次相识
- 消除误会
- 增加信任
- 推进暧昧关系
- 揭露某个隐瞒的信息

### 8.3 输出要求

模型输出需要稳定遵守以下约束：

- 不跳出当前地点与场景
- 不忽略当前出场角色设定
- 不把剧情直接推进到超出当前事件范围的阶段
- 保持现代恋爱叙事语气
- 兼顾旁白、角色反应与局势变化

## 9. 上下文与记忆系统

### 9.1 全局故事状态

全局故事状态在整局游戏中持续存在，不因为切场景或退出事件而清空。它保存：

- 主角当前处境
- 已认识的人物
- 人物关系变化
- 已发生的重要事件
- 当前未解决的情感线索
- 玩家的一般互动风格

### 9.2 当前事件上下文

每个剧情事件维护一份短期上下文，包含：

- 当前地点
- 当前角色
- 事件开场文本
- 最近若干轮对话
- 当前情绪焦点

### 9.3 原始聊天记录策略

- 不将整局原始对话无限累积给模型
- 只保留当前事件所需的近期对话作为短期上下文
- 通过手动压缩将长期信息转为摘要与关键记忆

### 9.4 手动记忆压缩

玩家可点击“记忆压缩”按钮，生成两部分内容：

1. 当前局势摘要
   - 简要概括最近剧情、关系状态和未解决问题
2. 关键记忆条目
   - 记录已知事实、关系变化、当前目标和玩家风格

### 9.5 手动压缩的原因

- 玩家更有参与感
- 避免自动压缩在错误时机打断沉浸
- 第一版实现更稳、更容易验证体验

### 9.6 模型调用的上下文拼装顺序

每次事件对话建议按以下顺序组织上下文：

1. 系统设定
2. 全局故事状态
3. 当前事件信息
4. 当前事件最近几轮对话
5. 玩家本轮输入

## 10. 技术方案

### 10.1 技术选型

- 前端：Vite + TypeScript
- 游戏展示层：Phaser
- UI 层：DOM 覆盖在游戏画面上

### 10.2 选型原因

- 项目属于 2D、地图点击、场景切换的移动端网页游戏，适合 Phaser
- 文本框、输入框和按钮更适合使用 DOM 进行开发与维护
- Phaser 负责上半屏的地图与主视觉，DOM 负责下半屏交互，有利于降低复杂度

### 10.3 状态管理策略

第一版使用轻量本地状态管理即可，不引入复杂状态方案。状态重点包括：

- 当前地图层级
- 当前地点
- 当前事件
- 事件完成记录
- 全局故事状态
- 当前事件上下文

### 10.4 模型接入策略

- 先设计统一的对话调用接口
- 早期开发先用 mock 事件回复完成流程验证
- 在交互链路稳定后再接入真实大模型 API

## 11. 系统模块划分

第一版建议拆分为以下模块：

### 11.1 地图模块

- 世界地图显示
- 区域点击
- 子场景切换

### 11.2 事件模块

- 事件触发判断
- 进入事件
- 结束事件
- 事件完成状态记录

### 11.3 对话模块

- 展示旁白与角色文本
- 接收玩家自然语言输入
- 拼装模型上下文
- 返回剧情文本

### 11.4 记忆模块

- 保存局势摘要
- 保存关键记忆
- 保存当前事件短期上下文
- 执行手动压缩

### 11.5 UI 模块

- 下半屏对话框
- 状态条
- 操作按钮
- 输入区

## 12. 推荐开发顺序

### 12.1 第一阶段：搭建基础外壳

- 建立手机竖屏布局
- 建立上下分屏结构
- 建立城市地图与区域点击
- 建立子场景切换
- 建立下半屏基础 UI

### 12.2 第二阶段：实现假事件流程

- 点击地点后显示进入提示
- 进入后展示固定开场剧情
- 玩家输入自然语言
- 系统返回 mock 剧情文本
- 允许结束事件并返回探索

### 12.3 第三阶段：接入记忆结构

- 维护全局故事状态
- 维护当前事件上下文
- 支持手动生成摘要与关键记忆
- 确认切场景后故事状态不丢失

### 12.4 第四阶段：接入真实模型

- 用真实 API 替换 mock 回复
- 调整上下文拼装方式
- 调整输出风格与稳定性

## 13. 验收标准

第一版完成时，应至少满足以下标准：

- 玩家能够在手机竖屏布局中完成一轮完整体验
- 玩家能够从城市地图进入主区域，再进入子场景
- 玩家能够在子场景中进入剧情事件
- 玩家能够在事件中用自然语言推进剧情
- 事件结束后，故事状态与记忆仍然保留
- 玩家能够手动触发记忆压缩，并看到摘要与关键记忆结果

## 14. 风险与控制策略

### 14.1 风险：内容范围膨胀

控制策略：

- 固定第一版地点数量
- 固定第一版角色数量
- 固定第一版事件数量

### 14.2 风险：模型输出跑偏

控制策略：

- 强化当前事件上下文约束
- 限制输出围绕当前场景目标展开
- 先用 mock 数据验证交互节奏

### 14.3 风险：产品退化成纯聊天

控制策略：

- 保持清晰的地图层和事件入口
- 确保每段剧情都来自明确地点与事件触发
- 保留“进入”和“退出”事件的游戏边界

## 15. 总结

第一版的产品定义是：

一个手机竖屏的现代恋爱向网页游戏。玩家在城市地图中探索地点，在具体子场景中触发剧情事件，并通过自然语言与角色互动推进关系。游戏通过全局故事状态和玩家手动记忆压缩机制，维持长剧情对话的连续性。

第一版的成功标准不是内容规模，而是验证以下三件事是否成立：

- 地图探索进入事件的手感成立
- 自然语言恋爱互动有沉浸感
- 记忆压缩足以支撑剧情连续推进
